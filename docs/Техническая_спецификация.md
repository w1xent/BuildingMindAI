# Техническая спецификация — BuildingMind AI MVP

## Архитектура системы

```
┌─────────────────────────────────────────────────────────────┐
│                        FRONTEND                             │

           │ Adapters
┌──────────▼──────────────────────────────────────────────────┐
│               Integration Layer                             │
│        BMS / HVAC / IoT Sensors / Energy Meters            │
│              (MQTT, Modbus, OPC-UA, HTTP)                   │
```


### Основные сервисы

**Problem Service** — главный сервис MVP:
- Хранит список активных проблем
- Принимает результаты AI-анализа
- Управляет статусами (active → in_progress → resolved)
- Отдаёт проблемы, отсортированные по приоритету

**Alert Service**:
- Принимает события от инженерных систем
- Дедублицирует похожие события
- Передаёт в AI/ML Service для анализа

**Auth Service**:
- JWT-авторизация
- Роли: facility_manager, chief_engineer, admin

### Модель данных

```sql
-- Проблема
CREATE TABLE problems (
    id          UUID PRIMARY KEY DEFAULT gen_random_uuid(),
    system_id   VARCHAR(100) NOT NULL,   -- "hvac_17", "ventilation_b"
    system_name VARCHAR(255) NOT NULL,
    description TEXT NOT NULL,
    priority    VARCHAR(10) NOT NULL,    -- high / medium / low
    status      VARCHAR(20) NOT NULL DEFAULT 'active',
    metric      VARCHAR(50),             -- "+22%"
    metric_label VARCHAR(100),
    why         TEXT,                    -- объяснение критичности
    cause       TEXT,                    -- возможная причина
    cause_probability VARCHAR(20),       -- high / medium / low
    consequences TEXT,
    detected_at TIMESTAMPTZ NOT NULL DEFAULT NOW(),
    updated_at  TIMESTAMPTZ NOT NULL DEFAULT NOW(),
    resolved_at TIMESTAMPTZ
);

-- Рекомендации (шаги)
CREATE TABLE recommendations (
    id          UUID PRIMARY KEY DEFAULT gen_random_uuid(),
    problem_id  UUID REFERENCES problems(id),
    step_order  INT NOT NULL,
    step_text   TEXT NOT NULL
);

-- Телеметрия (time-series)
CREATE TABLE telemetry (
    time        TIMESTAMPTZ NOT NULL,
    system_id   VARCHAR(100) NOT NULL,
    metric_name VARCHAR(100) NOT NULL,
    value       DOUBLE PRECISION NOT NULL,
    unit        VARCHAR(20)
);
SELECT create_hypertable('telemetry', 'time');

-- Пользователи
CREATE TABLE users (
    id       UUID PRIMARY KEY DEFAULT gen_random_uuid(),
    email    VARCHAR(255) UNIQUE NOT NULL,
    password_hash VARCHAR(255) NOT NULL,
    role     VARCHAR(50) NOT NULL DEFAULT 'facility_manager',
    name     VARCHAR(255)
);
```

### Логика MVP (упрощённая)

**Шаг 1 — Детекция аномалии:**
```python
# Сравнение текущего значения с нормой (baseline)
# MVP: статические пороги из конфига, позже — ML-модели
def detect_anomaly(metric_name, current_value, baseline):
    deviation = (current_value - baseline) / baseline * 100
    if abs(deviation) > threshold[metric_name]:
        return Anomaly(metric=metric_name, deviation=deviation)
```

**Шаг 2 — Приоритизация:**

## Принципы разработки

1. **API-first** — сначала контракт API, потом реализация
2. **Моки для быстрого старта** — до подключения реальных систем используем фиксированные данные
3. **Правило одного экрана** — каждое решение проверяем: помогает ли это в ядровом сценарии?
4. **Логирование событий** — все входящие события и решения системы пишем в лог для последующего анализа
5. **Безопасность** — JWT с коротким TTL (15 мин access + 7 дней refresh), HTTPS обязателен
